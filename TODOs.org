* Vision
*Sheaf* — a data-oriented configuration system for Hammerspoon automation.

Users configure their automation using orthogonal, composable concepts:
- *Events* - facts about something that happened, organized in a kind hierarchy
- *Commands* - named actions that can be invoked on a component
- *Components* - context-blind units with lifecycle (start/stop), state, and config
- *Behaviors* - component-blind rules mapping events to commands via a handler function
- *Tags* - contextual labels attached to component instances
- *Subscriptions* - the only place components, behaviors, and tags meet; the wiring that creates a system

See DESIGN.md for the full design reference.

The system should be:
- *Declarative* - configuration as data, not imperative code
- *Visualizable* - can see what's configured and what's happening
- *Debuggable* - can trace why something did or didn't trigger
- *Queryable* - can inspect state (via SQLite, REPL, UI)

**** DONE [#A] Decide routing rule storage strategy                     :architecture:
     Decision: behavior inputs via tagged component state.

     Routing rules are owned by a dedicated component, not by the behavior
     module, a separate ambient config module, or the subscription. The
     component owns the rules in its state and declares traits describing
     its attrs. A shape composes those traits for behavior input matching.

     Behaviors declare the component-state inputs they need:

     #+begin_src fennel
     (make-behavior
      {:name :url-dispatch.behaviors/route-url
       :respond-to [:event.kind.url/opened]
       :commands {:open-in-app :url-dispatch.commands/open-in-app
                  :show-chooser :url-dispatch.commands/show-chooser}
       :inputs {:rules :shape/url-routing-rules}
       :fn (fn [event candidates send-cmd inputs]
             ;; inputs.rules is raw state from the tagged routing-rules component
             ...)})
     #+end_src

     Subscriptions remain routing/wiring only. They select the event source,
     command target, and input component via tags:

     #+begin_src fennel
     {:behavior :url-dispatch.behaviors/route-url
      :source-tag :tag/url-handler
      :target-tag :tag/url-dispatch
      :input-tag :tag/url-routing-rules
      :event-selector :event.kind.url/opened}
     #+end_src

     The dispatcher resolves :input-tag to input candidates, validates that
     the selected component conforms to the behavior's declared input shape,
     and invokes the behavior as (fn [event candidates send-cmd inputs]).
     No separate cardinality field: the shape describes conformance, and tag
     assignment controls which components participate.

     For now inputs pass raw component state. Later, add trait-projected or
     read-only protected inputs so behaviors only see the fields guaranteed
     by the requested traits.

**** DONE [#A] Decide pattern matching syntax for URL rules             :architecture:
     Decision: structured URL matching with simple wildcard strings.

     Rules match against parsed URL fields rather than the full URL string.
     This keeps routing data serializable and readable while avoiding Lua
     pattern escaping as the primary user-facing syntax.

     #+begin_src fennel
     {:id :work-github
      :match {:urls [{:scheme [:http :https]
                      :host "*.github.com"
                      :path "/my-org/*"}]
              :sender-bundle-ids ["com.tinyspeck.slackmacgap"]}
      :action {:type :open-in-app
               :browser-id :chrome}}
     #+end_src

     URL match semantics:
     - [X] :scheme matches one scheme or a list of schemes
     - [X] :host supports exact host or leading wildcard subdomain match
           ("github.com", "*.github.com")
     - [X] :path supports "*" wildcards
     - [X] Missing URL fields mean "any"
     - [X] Multiple :match.urls entries are ORed
     - [X] Different keys inside one :match map are ANDed
     - [X] Rules are evaluated first-match-wins

     Sender match semantics:
     - [X] Match sender primarily by bundle ID, not app name
     - [X] Keep sender app name in events for display/debugging only

     Action semantics:
     - [X] Single target app: {:type :open-in-app :browser-id :safari}
     - [X] Multiple target apps / subset chooser:
           {:type :choose :browser-ids [:safari :chrome]}
     - [X] Predicate-function rules are deferred as a future escape hatch,
           not part of the initial data format

     Decoders are ordered URL wrapper transformers. They use structured
     match data, aligned with routing-rule URL matching, and keep only the
     service-specific extraction logic in code.

     #+begin_src fennel
     {:name :outlook-safelinks
      :match {:host "*.safelinks.protection.outlook.com"}
      :decode-fn (fn [ctx] ...)}
     #+end_src

     Decoder context:

     #+begin_src fennel
     {:url current-url
      :parts (hs.http.urlParts current-url)}
     #+end_src

     Pipeline semantics:
     - [X] Ordered decoders run against the current URL

** PROJ Sheaf Tiling Engine [0/8]
Niri-style scrolling column tiler built on Sheaf primitives.
Replaces paper-wm.fnl's monolithic tiling with composable behaviors and commands.
Windows arranged in columns on an infinite horizontal strip per screen;
viewport scrolls to keep the focused column visible.
Depends on: window-state component (done), window-watcher event source (done),
tag system (done), dispatcher (done).

*** LATER [#A] Design tiling data model :architecture:
    Core state for a tiling-state component. Research from Niri, PaperWM, Karousel
    shows all production scrolling tilers converge on the same structure.

    - [ ] Column list: ordered sequential array of columns
          Column = {:windows [window-id ...] :width N :active-window-idx N}
    - [ ] Reverse index: {window-id → {:col-idx N :row-idx N}} for O(1) lookup
    - [ ] Viewport offset: scalar tracking which slice of the strip is visible
    - [ ] Active column index: which column has focus
    - [ ] Decide width model: preset proportions (Niri: 1/3, 1/2, 2/3) vs
          content-driven (PaperWM: use window's natural width) vs configurable list
    - [ ] Decide per-column vs per-window width storage
    - [ ] Define traits for tiling state

*** LATER [#A] Implement column position computation :feature:
    Sequential accumulation: col_x[i] = col_x[i-1] + width[i-1] + gap.
    Pure function of state — no mutation during layout. All production tilers
    (Niri, PaperWM, Karousel) use this algorithm; O(n) per relayout.

    - [ ] Implement column-xs: state → [{:col-idx N :x N :width N} ...]
    - [ ] Implement within-column vertical tiling:
          single window gets full height; multiple windows split evenly
          (Karousel's fillSpace fencepost algorithm is a future refinement)
    - [ ] Implement tile-all!: given column-xs + viewport offset, call
          hs.window:setFrame on each window
    - [ ] Handle screen margins and window gaps from config
    - [ ] Command: :tiling.commands/retile — recomputes and applies layout

*** LATER [#A] Implement viewport scrolling :feature:
    Viewport offset determines which portion of the infinite strip is visible.
    Three strategies observed in production tilers; implement as swappable behaviors.

    - [ ] Minimum-scroll (Niri "never", Karousel lazy):
          scroll just enough to make focused column fully visible at nearest edge
    - [ ] Center-focused (Niri "always", Karousel centered):
          always center the active column in viewport
    - [ ] Overflow-center (Niri "on-overflow"):
          center only when focused + neighbor can't both fit; otherwise minimum-scroll
    - [ ] Implement as separate behaviors behind tags for easy swapping
    - [ ] Command: :tiling.commands/set-view-offset

*** LATER [#A] Implement window add/remove :feature:
    Niri stability invariant: adding/removing a window never resizes existing

    - [ ] Command: :tiling.commands/add-window — insert column at position

    - [ ] Command: :tiling.commands/remove-window — remove from column,

    - [ ] Apply position deltas to columns on the far side from focus

    - [ ] Commands: swap-left, swap-right, swap-up, swap-down

    User-adjustable column widths with preset cycling (Niri-style).

*** LATER [#B] Solve convergence on restart/reload :architecture:
    When Hammerspoon restarts or config reloads, windows exist at arbitrary
    positions. The tiler must infer the correct column structure from current
    window geometry and converge to a valid tiled layout with minimum disruption.
    Simple accumulation fails here — it ignores current positions and retiles
    in enumeration order, effectively rearranging all windows.

    - [ ] Approach A — spatial inference: sort windows by x-position, cluster
          into columns by proximity, then accumulate. Simple, ~30 LOC.
    - [ ] Approach B — Cassowary constraint solver: each window gets a weak
          "stay at current position" constraint plus required ordering/gap
          constraints. Solver finds minimum-disruption valid tiling.
          Most principled, ~1500 LOC for solver or use a library.
    - [ ] Approach C — persist column assignments to disk, restore on reload.
          Avoids inference entirely but adds I/O dependency.
    - [ ] Evaluate: is spatial inference good enough, or does Cassowary's
          minimum-disruption property justify the complexity?
    - [ ] If Cassowary: implement or port a Lua solver (kiwi-solver is ~1500 LOC C++)
    - [ ] Implement as a separate initialization behavior, swappable via tags

*** LATER [#C] Algorithm swapping via tags :architecture:
    Multiple tiling algorithms (or algorithm variants) can coexist as different
    behavior sets. Tags select which algorithm is active per screen/workspace.

    - [ ] Define tag convention: :tag/tiling-algo-accumulate, :tag/tiling-algo-cassowary
    - [ ] Behaviors for each algorithm variant share the same commands but differ
          in how they compute layout
    - [ ] Subscriptions wire the active algorithm's behaviors; swapping tags
          at runtime switches algorithms without restart
    - [ ] Useful for A/B testing scroll strategies, width policies, convergence methods
