* Vision
*Sheaf* — a data-oriented configuration system for Hammerspoon automation.

Users configure their automation using orthogonal, composable concepts:
- *Events* - facts about something that happened, organized in a kind hierarchy
- *Commands* - named actions that can be invoked on a component
- *Components* - context-blind units with lifecycle (start/stop), state, and config
- *Behaviors* - component-blind rules mapping events to commands via a handler function
- *Tags* - contextual labels attached to component instances
- *Subscriptions* - the only place components, behaviors, and tags meet; the wiring that creates a system

See DESIGN.md for the full design reference.

The system should be:
- *Declarative* - configuration as data, not imperative code
- *Visualizable* - can see what's configured and what's happening
- *Debuggable* - can trace why something did or didn't trigger
- *Queryable* - can inspect state (via SQLite, REPL, UI)

***** Expand traits to events and behaviors
      - Event schemas expressed as traits describing their data shape
      - Behaviors declare :requires-event-traits instead of :respond-to
      - Unifies all schema validation under one concept

***** Trait schema inheritance via hierarchy
      - Traits already support multiple parents via lib/hierarchy.fnl derive!
      - Add schema inheritance: a trait inherits all schema keys from its ancestors
      - Concrete trait schema = merge of ancestor schemas + own schema (child overrides parent on conflict)
      - Implement resolve-schema: given a trait, walk ancestors and merge schemas bottom-up
      - Implement serialize-trait: returns full trait description including inherited schema keys and hierarchy path

***** Migrate trait schemas from predicate functions to keyword descriptors (malli-inspired)
      - Current: {:menubar (fn [v] (not= nil v))} — predicate functions
      - Target: {:menubar :non-nil} or {:menubar [:map {:title :string}]} — keyword-based schemas
      - Inspired by Clojure Malli: composable schema descriptors instead of opaque functions
      - Benefits: serializable, inspectable, self-documenting, can generate validators
      - Define a schema registry mapping keywords to validator fns (like malli's registry)
      - Built-in descriptors: :non-nil, :string, :number, :table, :boolean, :function
      - Composite descriptors: [:vector :string], [:map {:key :type}], [:enum :a :b :c]
      - Evaluate other malli/spec ideas:
        - Schema transformers (coerce, decode, encode)
        - Schema generation (for testing/fuzzing)
        - Human-readable error messages from schema validation failures
        - Registry of reusable schema fragments

***** Higher-level schema concept (role-based component profiles)
      - A schema bundles multiple traits into named profiles describing component roles
      - Example: (make-schema :component-profiles
                   {:ui-indicator [:trait/has-menubar :trait/has-delayed-timer]
                    :window-manager [:trait/has-window-filter :trait/has-layout]
                    :simple-trigger []})
      - A component satisfying all traits in a profile is said to "conform" to that profile
      - Use cases: validate component type conforms to a profile at registration time,
        commands can require a profile instead of listing individual traits,
        documentation and introspection ("this component is a ui-indicator")
      - Profiles can compose: {:full-ui [:ui-indicator :interactive]} where :interactive
        is itself a profile
      - Inspired by Clojure spec's s/keys with :req and :opt, and malli's :multi schemas

***** Traits as datalog schemas
      - State migrates from Lua tables to a datalog database
      - Traits become datalog schemas — sets of attribute definitions
      - Query component state via datalog queries instead of direct table access

***** Commands as transactions
      - Commands return facts (assertions/retractions) instead of new state
      - Dispatcher applies the transaction to the DB
      - Enables undo, history, time-travel debugging
      - Aligns with Datomic inspiration in DESIGN.md

***** Sub-components (component instance tree)
      - Components can instantiate and own child components
      - Parent lifecycle controls children: stopping parent stops children
      - Enables paper-wm component owning per-window child components

**** DONE [#A] Decide routing rule storage strategy                     :architecture:
     Decision: behavior inputs via tagged component state.

     Routing rules are owned by a dedicated component, not by the behavior
     module, a separate ambient config module, or the subscription. The
     component owns the rules in its state and declares a trait describing
     the shape, e.g. :trait/has-url-routing-rules.

     Behaviors declare the component-state inputs they need:

     #+begin_src fennel
     (make-behavior
      {:name :url-dispatch.behaviors/route-url
       :respond-to [:event.kind.url/opened]
       :commands {:open-in-app :url-dispatch.commands/open-in-app
                  :show-chooser :url-dispatch.commands/show-chooser}
       :inputs {:rules [:trait/has-url-routing-rules]}
       :fn (fn [event candidates send-cmd inputs]
             ;; inputs.rules is raw state from the tagged routing-rules component
             ...)})
     #+end_src

     Subscriptions remain routing/wiring only. They select the event source,
     command target, and input component via tags:

     #+begin_src fennel
     {:behavior :url-dispatch.behaviors/route-url
      :source-tag :tag/url-handler
      :target-tag :tag/url-dispatch
      :input-tag :tag/url-routing-rules
      :event-selector :event.kind.url/opened}
     #+end_src

     The dispatcher resolves :input-tag to component state, validates that
     the selected component satisfies the behavior's declared input traits,
     and invokes the behavior as (fn [event candidates send-cmd inputs]).
     No separate cardinality field: the trait describes the state shape, and
     tag assignment controls which component participates.

     For now inputs pass raw component state. Later, add trait-projected or
     read-only protected inputs so behaviors only see the fields guaranteed
     by the requested traits.

**** NEXT [#A] Decide pattern matching syntax for URL rules             :architecture:
     Rule = {:patterns [...]} — but what syntax and what does it match against?

     Options:
     - A: Lua patterns against full URL — built-in, no deps, limited (no alternation)
     - B: String prefix/contains matching — simplest, covers most cases (e.g. "github.com")
     - C: Structured matching — separate fields {:host "pattern" :path "pattern" :scheme "pattern"}
     - D: Predicate functions — most flexible, not serializable

     - [ ] Decide whether :sender matches app name, bundle ID, or both
     - [ ] Decide whether one rule can target multiple apps and show chooser
           among that subset

**** NEXT [#A] Decide decoder pipeline contract                         :architecture:
     Current schema Decoder = {:name :match :decode-url?} is ambiguous.

     Clarify:
     - [ ] :match — predicate fn (fn [url] → bool)? Or data pattern against host/URL?
     - [ ] :decode-url? — name suggests predicate but should be transformer.
       Actual need: (fn [url] → decoded-url-or-nil)
     - [ ] Chain semantics: when decoder returns nil, pass through unchanged?
           Or stop chain?

     Suggested schema (decide between):
     - A: {:name string :match-fn (fn [url] → bool) :decode-fn (fn [url] → url)}
       Two functions — explicit match gate, separate transform
     - B: {:name string :decode-fn (fn [url] → url-or-nil)}
       Single function — nil return means "didn't match, pass through"
     - C: {:name string :host-pattern string :decode-fn (fn [url] → url)}
       Data-oriented match (Lua pattern on host), function transform

**** NEXT [#A] Decide fallback and chooser browser list strategy        :architecture:
     When no routing rule matches, what happens?

     Options for fallback:
     - A: Always show chooser — safe but annoying for frequent use
     - B: Configurable default browser bundle ID — practical for daily use,
       chooser only on explicit fallback rule
     - C: Both — default browser + hotkey to force chooser

     Where does the chooser's browser list come from?
     - A: Hardcoded list of known browsers
     - B: Discovered via hs.application.infoForBundleID for a configured list
     - C: All installed apps that handle http URLs (may be noisy)
     - [ ] Decide fallback behavior when no rule matches
     - [ ] Decide source of chooser browser list
